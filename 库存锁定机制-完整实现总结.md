#### 3.2 Service 实现
**ScheduleMaterialLockingServiceImpl.java** (389 行)

**核心算法流程**:
```
@Transactional lockMaterialsForSchedule(Long scheduleId)
  ├─ 1. 查询排程信息和订单列表
  │
  ├─ 2. FOR EACH 订单:
  │   ├─ 第一步: 查询可用复卷（层级1）
  │   │   └─ lockFromReels("复卷", requiredArea, allocatedArea)
  │   │       ├─ 构建查询条件 (厚度/宽度/最小面积)
  │   │       ├─ 遍历物料卷
  │   │       ├─ 乐观锁更新库存 (version 防冲突)
  │   │       └─ 创建锁定记录
  │   │
  │   ├─ 第二步: 复卷不足，查询母卷（层级2）
  │   │   └─ 同上流程
  │   │
  │   ├─ 第三步: 母卷不足，触发涂布排程（层级3）
  │   │   ├─ 创建 ScheduleMaterialAllocation 记录
  │   │   ├─ 标记 needCoating = true
  │   │   └─ 返回 needCoatingOrderIds
  │   │
  │   └─ 统计本订单锁定结果
  │
  ├─ 3. 汇总统计
  │   ├─ 完全锁定订单数
  │   ├─ 部分锁定订单数
  │   ├─ 未锁定订单数
  │   ├─ 总面积/分配面积
  │   └─ 需要涂布的订单IDs
  │
  └─ 4. 返回 LockResult 结果

@Transactional allocateLocks(List<Long> lockIds)
  ├─ 1. 批量更新锁定状态为 "已领料"
  ├─ 2. 更新库存消耗面积 (updateConsumedArea)
  │   └─ 乐观锁版本号检查
  └─ 3. 记录领料时间和操作人

@Transactional releaseLocks(Long scheduleId)
  ├─ 1. 查询排程所有锁定记录
  ├─ 2. 批量更新状态为 "已释放"
  ├─ 3. 恢复库存
  │   └─ 减少预留面积，增加可用面积
  └─ 4. 记录释放时间
```

**关键特性**:
- ✅ **三层级物料查询**: 复卷 → 母卷 → 涂布
- ✅ **乐观锁并发控制**: 版本号 + UPDATE WHERE version = ?
- ✅ **事务管理**: @Transactional(rollbackFor = Exception.class)
- ✅ **多对多关系**: 一卷多订单，一订单多卷
- ✅ **完整日志**: @Slf4j 记录每个步骤

## 📊 核心概念说明

### 库存锁定状态流转
```
┌──────────┐     排程启动      ┌──────────┐     生产领料    ┌──────────┐
│  未锁定  │  ─────────────→  │  锁定中  │  ─────────────→  │  已领料  │
└──────────┘                   └──────────┘                   └──────────┘
                                    │
                                    │ 排程取消
                                    ↓
                               ┌──────────┐
                               │  已释放  │
                               └──────────┘

┌──────────┐     生产反馈      ┌──────────┐
│  已领料  │  ─────────────→  │  已消耗  │
└──────────┘                   └──────────┘
```

### 面积计算公式
```
可用面积 (availableArea) = 总面积 - 已锁定面积 - 已消耗面积
已锁定面积 (reservedArea) = 被锁定物料的累计面积
已消耗面积 (consumedArea) = 已领料的累计面积
```

### 三层级物料查询
```
Level 1 (复卷)
  ├─ 查询条件: reel_type = '复卷' AND availableArea >= requiredArea
  ├─ 优先级: 最高
  └─ 缺点: 数量有限

Level 2 (母卷)
  ├─ 查询条件: reel_type = '母卷' AND availableArea >= requiredArea
  ├─ 优先级: 中等
  └─ 缺点: 需要复卷化处理

Level 3 (涂布)
  ├─ 查询条件: 触发涂布排程
  ├─ 优先级: 最低
  └─ 优点: 可以满足任何数量的需求
```

### 乐观锁并发控制
```sql
-- 更新物料库存时
UPDATE tape_stock
SET reserved_area = reserved_area + ?,
    version = version + 1
WHERE id = ? AND version = ?

-- 如果 affected_rows = 0，说明发生了并发冲突
-- 应该重试查询最新库存并重新锁定
```

## 🔧 故障排查指南

### 问题 1: 数据库表创建失败
**症状**: MySQL 错误信息显示外键约束失败
**解决**:
1. 检查 batch_schedule 和 sales_order 表是否存在
2. 检查字段类型是否匹配（都应该是 BIGINT）
3. 尝试先删除表再重新创建: `DROP TABLE IF EXISTS schedule_material_lock;`

### 问题 2: 物料锁定返回空结果
**症状**: lockMaterialsForSchedule 返回 0 个锁定记录
**解决**:
1. 检查排程是否关联了订单
2. 检查库存中是否有可用物料 (availableArea > 0)
3. 检查订单的厚度/宽度是否与库存匹配
4. 查看日志中的 DEBUG 信息

### 问题 3: 乐观锁冲突（更新失败）
**症状**: 某些物料锁定失败，affected_rows = 0
**解决**:
1. 这是正常的并发现象，实现中已处理
2. 检查是否有多个排程同时启动
3. 增加重试机制（最多重试 3 次）

### 问题 4: 前端页面未显示
**症状**: 访问物料锁定管理页面 404
**解决**:
1. 确认路由已配置: `/material-lock`
2. 确认 scheduleMaterial.js 已导入
3. 检查浏览器控制台错误信息
4. 确认后端 API 可正常访问
