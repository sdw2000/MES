## 二、功能验证步骤

### 2.1 访问订单预处理页面
1. 登录系统 (确保有 production 或 admin 角色)
2. 点击左侧菜单: 计划管理 → 订单预处理
3. 看到订单列表页面即表示功能正常

### 2.2 查询订单
```
输入搜索条件:
- 订单号: (可选) 输入订单号搜索
- 物料代码: (可选) 输入料号搜索
- 状态: (可选) 选择 "待预处理" / "已锁定" / "已派发"

点击"查询"按钮
```

### 2.3 编辑数量
```
在数量列输入框中修改订单需求面积(m²)
修改后自动更新
```

### 2.4 选择库存物料
```
1. 点击"库存选择"按钮 (如物料代码不为空)
2. 弹窗显示该物料的所有可锁定库存
3. FIFO顺序: 按生产日期从旧到新排列
4. 每行显示:
   - 二维码 (QR code/批次号)
   - 总卷数 / 已锁定卷数 = 可锁定卷数
   - 可用面积(m²)
5. 使用 Checkbox 多选要锁定的物料卷次
6. 点击"确认锁定"提交
```

### 2.5 提交订单
```
1. 确认已选择物料且排程类型正确
2. 点击"提交"按钮
3. 确认对话框中确认
4. 订单状态变为"已派发"
5. 订单进入相应的待排池:
   - 涂布 → pending_coating_order_pool
   - 复卷 → production_schedule
   - 分切 → production_schedule
```

## 四、常见问题排查

### Q1: 页面找不到或404
**原因**: 路由未正确配置或前端未更新
**解决**:
```bash
# 1. 检查路由配置
# src/router/index.js 中应该有:
{
  path: 'order-material-lock',
  component: () => import('@/views/production/orderPreprocessing'),
  name: 'OrderPreprocessing',
  meta: { title: '订单预处理', roles: ['production', 'admin'] }
}

# 2. 清理浏览器缓存
# F12 → Application → Clear site data

# 3. 重新启动前端
npm run dev
```

### Q2: 后端返回401(未授权)
**原因**: Token过期或权限不足
**解决**:
```bash
# 1. 重新登录获取新token
# 2. 确认用户角色是 production 或 admin
# 3. 检查后端权限配置
@PreAuthorize("hasAuthority('production')")
```

### Q3: 物料选择弹窗为空
**原因**: 查询不到该物料的库存
**解决**:
```bash
# 1. 验证物料代码是否存在
SELECT * FROM tape_stock WHERE material_code = 'FT-001' LIMIT 10;

# 2. 验证库存状态 (status=1表示有效)
SELECT * FROM tape_stock 
WHERE material_code = 'FT-001' AND status = 1;

# 3. 检查是否所有库存都被锁定
SELECT tape_stock_id, SUM(lock_area) FROM order_material_lock 
WHERE tape_stock_id IN (SELECT id FROM tape_stock WHERE material_code = 'FT-001')
GROUP BY tape_stock_id;
```

### Q4: 乐观锁冲突 ("物料库存已被其他订单修改")
**原因**: 多个用户同时锁定同一卷物料
**解决**:
```bash
# 1. 刷新页面重新查询
# 2. 重新选择物料
# 3. 可以选择其他可用卷

# 系统会自动处理冲突，无需手动干预
```

### Q5: 提交后订单未进入待排池
**原因**: 排程类型判断错误或目标池配置问题
**解决**:
```bash
# 1. 检查排程类型是否正确显示
# 2. 验证目标池配置
SELECT * FROM order_preprocessing WHERE id = ?;

# 3. 检查log文件查看错误信息
tail -f logs/mes.log | grep -i preprocessing
```

## 五、数据查询参考

### 5.1 查看所有待处理订单
```sql
SELECT id, order_no, order_item_code, material_code, required_qty, locked_qty, 
       lock_status, schedule_type, `status`, created_at
FROM order_preprocessing
WHERE `status` = 'preprocessing'
ORDER BY created_at DESC;
```

### 5.2 查看已锁定的物料
```sql
SELECT oml.id, oml.order_id, oml.batch_no, oml.lock_qty, oml.lock_area, 
       ts.material_code, ts.total_rolls
FROM order_material_lock oml
JOIN tape_stock ts ON oml.tape_stock_id = ts.id
WHERE oml.lock_status = 'locked'
ORDER BY oml.locked_at DESC;
```

### 5.3 统计一卷物料被锁定的情况
```sql
SELECT tape_stock_id, COUNT(*) as lock_count, 
       SUM(lock_area) as total_locked_area
FROM order_material_lock
WHERE lock_status = 'locked'
GROUP BY tape_stock_id;
```

### 5.4 查看已派发的订单
```sql
SELECT id, order_no, schedule_type, target_pool, 
       `status`, updated_at
FROM order_preprocessing
WHERE `status` = 'dispatched'
ORDER BY updated_at DESC;
```

## 六、性能建议

### 6.1 数据库优化
```sql
-- 创建索引优化查询性能
ALTER TABLE order_preprocessing 
ADD INDEX idx_order_no (order_no),
ADD INDEX idx_material_code (material_code),
ADD INDEX idx_status (`status`);

ALTER TABLE order_material_lock
ADD INDEX idx_preprocessing_id (preprocessing_id),
ADD INDEX idx_tape_stock_id (tape_stock_id),
ADD INDEX idx_fifo_order (fifo_order);
```

### 6.2 查询优化
- 预处理列表分页查询，单页不超过50条记录
- 物料列表FIFO查询限制返回50条
- 避免在循环中查询数据库

### 6.3 并发控制
- 使用乐观锁防止库存超卖
- 建议同一时间最多支持10个用户并发操作
- 高并发场景下考虑使用消息队列异步处理

## 八、维护任务

### 8.1 定期检查
```bash
# 1. 检查数据一致性
SELECT COUNT(*) FROM order_preprocessing WHERE `status` = 'preprocessing';

# 2. 检查锁定状态
SELECT COUNT(*) FROM order_material_lock WHERE lock_status = 'locked';

# 3. 检查库存reserved_area是否正确
SELECT id, reserved_area, (
  SELECT SUM(lock_area) FROM order_material_lock 
  WHERE tape_stock_id = tape_stock.id
) as calculated_reserved
FROM tape_stock;
```

### 8.2 数据清理
```bash
# 清理过期的已派发记录 (保留30天)
DELETE FROM order_preprocessing 
WHERE `status` = 'dispatched' AND updated_at < DATE_SUB(NOW(), INTERVAL 30 DAY);
```
