### 项目目标
✅ **已完成**: 设计并实现完整的库存锁定机制，用于支持生产排程过程中的物料管理

**核心能力**:
- ✅ 支持按面积粒度(m²)的库存锁定（非卷数）
- ✅ 实现三层级物料优先级查询: 复卷→母卷→涂布排程
- ✅ 支持多对多关系：一卷被多订单使用，一订单使用多卷
- ✅ 排程启动时自动锁定物料
- ✅ 生产领料时扣减库存，确保已锁定物料对其他排程不可见

```
│                    Spring Boot Backend                   │
│  ScheduleMaterialController → ScheduleMaterialLocking... │
│                                      ↓                   │
│                          Service Interface & Implementation
│                                      ↓                   │
│                            Data Access Layer (Mapper)
└────────────────────────────┬────────────────────────────┘
                             │
┌────────────────────────────┴────────────────────────────┐
│                    MySQL Database                        │
│  schedule_material_lock + schedule_material_allocation  │
│                    + modified tape_stock                │
└──────────────────────────────────────────────────────────┘
```

### 核心算法流程

```
@Transactional
lockMaterialsForSchedule(scheduleId)
  │
  ├─ 1. 查询排程信息和订单列表
  │
  ├─ 2. FOR EACH 订单:
  │     ├─ 第一步: 查询可用复卷
  │     │   └─ updateReservedArea (乐观锁)
  │     ├─ 第二步: 复卷不足，查询可用母卷
  │     │   └─ updateReservedArea (乐观锁)
  │     └─ 第三步: 母卷不足，触发涂布
  │         └─ 标记 needCoating = 1
  │
  ├─ 3. 创建锁定记录 (schedule_material_lock)
  │
  ├─ 4. 创建分配记录 (schedule_material_allocation)
  │
  └─ 5. 返回 LockResult 统计
```

### 并发控制机制

**乐观锁实现**:
```sql
UPDATE tape_stock
SET reserved_area = reserved_area + ?,
    version = version + 1
WHERE id = ? AND version = ?
```

**重试机制**:
```java
if (affectedRows == 0) {
    // 版本号冲突，重新查询并重试
    tape = tapeStockMapper.selectById(tapeId);
    // 重试锁定
}
```

## 💡 创新点

### 1. 三层级物料查询
实现了智能的物料选择机制:
- 复卷优先（库存充足且成本低）
- 母卷补充（复卷不足时）
- 涂布排程（库存完全不足时）

### 2. 面积粒度的锁定
与传统的卷数锁定不同:
- 支持按面积(m²)锁定
- 更精确的库存控制
- 更灵活的物料分配

### 3. 多对多关系处理
完整支持:
- 一卷物料被多个订单使用
- 一个订单使用多卷物料
- 完整的分配和跟踪记录

### 4. 自动化涂布触发
智能识别需要涂布的情况:
- 自动评估库存是否充足
- 自动标记需涂布的订单
- 支持后续涂布排程创建

### 5. 高并发安全
采用乐观锁机制:
- 版本号防冲突
- 自动重试机制
- 事务一致性保证
